A learning walkthrough

Building a Chrome extension, one honest layer at a time.

Eighteen parts through a real Manifest V3 extension written in TypeScript with React. We start from the manifest, descend into the service worker, the popup, the Chrome APIs, then rise back up through the architecture, the distributed sync model, the custom algorithms, and the testing harness. Read each part next to the file it cites — the tutorial is the map, the source is the territory.

Part 01 — Foundations

Anatomy of a Manifest V3 extension

A Chrome extension is a small web app Chrome lets run in privileged contexts. Everything starts from one declaration file: manifest.json. 

{
  "manifest_version": 3,
  "name": "Tab Group Sync",
  "version": "1.2.0",
  "permissions": ["tabs", "tabGroups", "bookmarks", "storage", "unlimitedStorage", "alarms"],
  "action":     { "default_popup": "popup.html" },
  "background": { "service_worker": "background.js", "type": "module" },
  "icons":      { "16": "icons/icon16.png", "48": "icons/icon48.png", "128": "icons/icon128.png" }
}
manifest.json — the extension's front door
  • manifest_version 3. The current model. MV2 is retired. The biggest MV3 change is replacing the persistent background page with an ephemeral service worker. That one word rewires the rest of the architecture — see Part 5.
  • permissions. Each string unlocks a Chrome API. tabGroups, bookmarks, alarms map 1:1 to chrome.tabGroups.*, chrome.bookmarks.*, chrome.alarms.*.
  • action.default_popup. The toolbar button opens this HTML as a small window — the popup.
  • background.service_worker. Runs outside any tab, can listen to global browser events, is terminated by Chrome when idle.

The three contexts

┌─────────────────────────┐ ┌──────────────────────────┐ │ Popup (React app) │◀────▶│ Service Worker (bg) │ │ • popup.html + popup.js│ port │ • background.js │ │ • lives while open │ │ • Chrome events, sync │ └─────────────────────────┘ └──────────┬───────────────┘ │ ▼ chrome.tabGroups / chrome.bookmarks / chrome.storage

There's a third context — content scripts that inject into web pages. This extension doesn't use any; it only manipulates Chrome-level state (groups, bookmarks, storage). You'll build simpler mental models if you ignore content scripts until you need them.

Key idea
Almost every extension bug, eventually, comes back to confusion about which context a line of code is running in, or the fact that the service worker was quietly terminated between two events.
Try this
Visit chrome://extensions, enable Developer mode, click Load unpacked, pick dist/. Pin the icon — that's the popup. Click "service worker" under the extension card — that's the background's DevTools console.
Part 02 — Foundations

Project scaffold and the dual build

TypeScript needs compilation. Chrome's service worker has a peculiar constraint: it must be a single file with all imports inlined. This project solves both with Vite running in two passes. 

// vite.config.ts
const buildBackground = process.env.BUILD_BACKGROUND === 'true';

export default defineConfig({
  plugins: [react()],
  build: {
    outDir: 'dist',
    emptyOutDir: !buildBackground,           // keep popup output on pass 2
    rollupOptions: {
      input: buildBackground
        ? { background: resolve(__dirname, 'src/background.ts') }
        : { popup:      resolve(__dirname, 'src/popup.tsx') },
      output: { inlineDynamicImports: buildBackground }, // bg = single file
    },
    target: 'chrome112',
  },
});
# package.json
"build": "tsc && vite build && BUILD_BACKGROUND=true vite build && node scripts/copy-extension-files.js"
  1. tsc — typecheck only (Vite does the transpile).
  2. First Vite pass — produces popup.js and its chunks.
  3. Second Vite pass (BUILD_BACKGROUND=true) — produces one inlined background.js.
  4. Copy script — moves manifest.json, HTML, icons into dist/.

TypeScript and the Chrome types

The @types/chrome package gives the Chrome APIs their type signatures, so chrome.tabGroups.query({}) autocompletes and returns a typed Promise<chrome.tabGroups.TabGroup[]>. This is one of the largest productivity wins you get from TypeScript in extension work — the Chrome API surface is enormous and documented primarily through clicks in developer.chrome.com.

Why Vite?
You could use webpack, esbuild, Parcel, or raw tsc. Vite's advantage here is fast rebuilds during npm run watch and a small config surface. Nothing in the extension model requires Vite specifically.
Try this
Run npm run build, inspect dist/. Open background.js — see how every src/lib/* import was inlined. Compare with popup.js, which uses chunks under assets/.
Part 03 — Foundations

Code organization

Good layout tells you where to put new code without thinking. This project separates bridges (listeners), domain logic (managers), UI (components), and configuration (specs + manifest). Dependencies flow one direction. 

src/
├── background.ts              # service worker entry — composition root
├── popup.tsx                  # popup entry — React bootstrap
├── listeners/                 # bridge: Chrome events → manager methods
│   ├── tabGroupListeners.ts
│   ├── tabListeners.ts
│   └── bookmarkListeners.ts
├── lib/                       # domain logic (no React, no DOM)
│   ├── storage/storageManager.ts
│   ├── bookmarks/             # bookmarkManager, bookmarkQueries/Mutations, snapshotManager
│   ├── sync/syncEngine.ts     # orchestrator
│   ├── tabGroupManager.ts
│   ├── types/                 # shared TypeScript types
│   └── utils/                 # logger, errors, validators, name resolver, windowLabelBuilder
└── components/                # React UI
    ├── App.tsx                # root; owns manager instances
    ├── GroupList.tsx  GroupSection.tsx
    ├── Settings.tsx  FolderPicker.tsx
    ├── MoveGroupDialog.tsx  SnapshotList.tsx
    └── ErrorBoundary.tsx

tests/
├── unit/        # Vitest, mocked chrome APIs
├── property/    # fast-check invariants
└── e2e/         # Playwright, real Chrome with extension loaded

.kiro/
├── specs/       # requirements.md, design.md, tasks.md per feature
└── steering/    # product / tech / structure guidance

Dependency direction

Chrome events ──▶ listeners/ ──▶ lib/ (managers) ──▶ chrome.* APIs ▲ │ components/ ─┘ (App.tsx owns manager instances; children receive them as props)

UI components depend on managers. Listeners depend on managers. Managers depend on each other (with one explicit circular reference patched at startup — Part 8). No manager imports React, the DOM, or a listener. That asymmetry is what lets us unit-test managers in Node with a mocked chrome global.

Specs as code

The .kiro/specs/ directory is the system of record. Each feature has requirements.md (what), design.md (how), and tasks.md (a checklist). Before writing code, the contract goes there. If scope changes, the spec changes first. This repo has four specs: tab-group-sync (core), sw-reliability, move-group-across-windows, and observability-openobserve.

Convention
When you can't decide where a new piece of code goes, ask: does it touch React? → components. Does it touch chrome.* events only? → listeners. Pure business logic? → lib. If three pieces live in different places, the bridge between them usually belongs in a manager.
Part 04 — Foundations

UI frameworks: React (and a look at Svelte)

The popup is a React app. If you've only done backend work, React's mental model can feel backwards. This is the minimum you need to read this codebase fluently — plus a sketch of how Svelte would solve the same problem differently.

Four React concepts

1. Components are functions that return JSX

export default function Header({ title }: { title: string }) {
  return <h1>{title}</h1>;
}

JSX is syntactic sugar: <h1>...</h1> compiles to React.createElement('h1', ...). A component is just a function that React calls to get a description of what the DOM should look like.

2. useState gives a component memory

const [expanded, setExpanded] = React.useState(true);
// later:
<button onClick={() => setExpanded(!expanded)}>Toggle</button>

Calling setExpanded schedules a re-render. React re-invokes the component function, sees the new JSX, and updates the minimum DOM needed. You never write element.textContent = ... — you declare what the UI should look like for a given state.

3. useEffect runs side effects after render

React.useEffect(() => {
  const port = chrome.runtime.connect({ name: 'popup' });
  port.postMessage({ type: 'PING' });
  return () => port.disconnect();    // cleanup on unmount or before next run
}, []);   // [] = run once on mount

The second argument controls when the effect re-runs: [] means once; [x, y] means whenever x or y changes; omitted means every render (rare).

4. Props are function arguments, typed as interfaces

interface GroupSectionProps {
  title: string;
  groups: GroupViewModel[];
  storage: StorageManager;
  onToggleSync: (name: string) => void;     // callback passed from parent
  readOnly?: boolean;                        // ? = optional
}

export default function GroupSection(props: GroupSectionProps) { /* ... */ }

Data flows down (parent → child via props). Events flow up (child → parent via callback props). This is the unidirectional data flow you'll hear about. It makes the app easier to reason about than two-way data binding.

Svelte, as a comparison

Svelte reaches the same UI-rebuilding-from-state outcome with different ergonomics — compile-time reactivity instead of runtime diffing:

<script lang="ts">
  let expanded = true;     // reactive "cell" — just a variable
  $: label = expanded ? 'Collapse' : 'Expand';   // $: = recompute on deps change
</script>

<button on:click={() => expanded = !expanded}>{label}</button>
ReactSvelte
ReactivityRuntime diff via virtual DOMCompile-time, mutates DOM directly
StateuseState hook returns pairTop-level let
EffectsuseEffect(fn, deps)$: label & onMount
Bundle sizeLarger (React runtime + MUI)Smaller (compiled away)
EcosystemEnormous component libsSmaller, younger

This project chose React for library ecosystem (MUI gives us switches, dialogs, tree views that look native inside Chrome) and for familiarity. Svelte would be a fine choice for a simpler popup; the mental model changes (fewer hooks, more plain variables) but the Chrome-side code in this tutorial is identical either way.

Key idea
UI frameworks solve one problem: keep the DOM in sync with state, without forcing you to imperatively move bytes. Once you understand that, the specific syntax is a detail.
Part 05 — Runtime

The service worker — a background that forgets

The service worker is where your extension thinks. It also dies unpredictably. Those two facts produce most of the interesting code in src/background.ts.

Chrome terminates the worker after ~30 s of no events. When a new event fires, Chrome spins a fresh worker, re-runs background.js top-to-bottom, then dispatches the event. In-memory state is gone. This is the hardest mental shift from MV2.

Walkthrough: ensureInitialized

Every listener in background.ts calls this before touching a manager. Four responsibilities in twelve lines — the hot-path short-circuit, concurrency guard, failure containment, and retry reset.

// src/background.ts
let isReady = false;
let initPromise: Promise<boolean> | null = null;

async function ensureInitialized(): Promise<boolean> {
  if (isReady) return true;                                      // ①
  if (initPromise) return initPromise;                           // ②
  initPromise = (async () => {                                   // ③
    try {
      await initializeManagers();
      return true;
    } catch (error) {
      logger.error('ensureInitialized:failed', { error: /*…*/ });
      return false;                                              // ④
    }
  })().then(result => { initPromise = null; return result; });   // ⑤
  return initPromise;
}
  1. Hot path. Once initializeManagers() has set isReady = true, every subsequent call is a single boolean read. Zero cost, no microtask. This matters because every message handler starts with await ensureInitialized().
  2. Concurrency guard. If two events fire on a fresh worker at the same moment, the second caller sees initPromise already exists and awaits it. Without this line you'd double-init managers and double-register listeners.
  3. IIFE wrapped in a shared promise. We start the async work immediately and assign the resulting promise to a module-level variable before any await. That's what makes step ② observable to concurrent callers.
  4. Failure returns false, not throws. Handlers then do if (!await ensureInitialized()) return — a no-op is safer than a crash that might poison the event listener for the rest of the worker's life.
  5. Clear on settle. Whether init succeeded or failed, initPromise is reset to null. If it failed, the next event will retry from scratch. If it succeeded, isReady is now true so step ① wins anyway.

chrome.alarms, not setInterval

setInterval is dead weight in MV3 — the worker dies and takes it with it. Use chrome.alarms: Chrome persists them and fires on your worker, waking it if needed. 

await chrome.alarms.create('periodic-sync', { periodInMinutes: 5 });

chrome.alarms.onAlarm.addListener(async (alarm) => {
  if (alarm.name !== 'periodic-sync') return;
  if (!await ensureInitialized()) return;        // wake + init if cold
  await syncEngine.syncAll();
});
Rule of thumb
Every listener in the service worker should assume "I might be the first thing running in a fresh worker." That's why ensureInitialized() is called at the top of each handler.
Part 06 — Runtime

Listening to browser events

An extension is mostly a bag of event listeners. Chrome fires events, your code reacts. Below is the pattern used throughout src/listeners/.

Walkthrough: processGroupQueue

When Chrome restores a window, onCreated fires many times within milliseconds. Parallel handlers would race on the same bookmark folder. The queue serializes with a gap between items — backpressure in a handful of lines.

// src/listeners/tabGroupListeners.ts
export function initializeTabGroupListeners(tabGroupManager: TabGroupManager): void {
  let groupQueue: chrome.tabGroups.TabGroup[] = [];              // ①
  let processingQueue = false;

  async function processGroupQueue() {
    if (processingQueue || groupQueue.length === 0) return;      // ②
    processingQueue = true;
    try {
      const group = groupQueue.shift()!;                         // ③
      await tabGroupManager.handleGroupVisible(group);
    } catch (error) {
      logger.error('tabGroup:queue:failed', { /*…*/ });          // ④
    } finally {
      processingQueue = false;
      if (groupQueue.length > 0) {
        setTimeout(processGroupQueue, 1000);                     // ⑤
      }
    }
  }

  chrome.tabGroups.onCreated.addListener((group) => {
    groupQueue.push(group);
    processGroupQueue();                                         // ⑥ fire-and-forget
  });
}
  1. Closure-local state. The queue and flag live inside initializeTabGroupListeners, not at module scope. That keeps them private to these listeners and invisible to the rest of the worker.
  2. Double-entry guard. Any call made while we're already draining is a no-op — the existing drain will pick up the newly pushed item. This is the line that makes the queue serial.
  3. One item per invocation. We handle a single group, not a while loop. Pairing this with step ⑤ gives each item its own setTimeout slot, which lets the service worker's event loop breathe between items.
  4. Error isolation. A failed group doesn't abort the queue. The catch logs and falls through to finally, which resets the flag and reschedules.
  5. Self-rescheduling. When more items are pending, a 1-second setTimeout queues the next drain. That gap gives Chrome time to settle, prevents quota bursts, and yields the worker to unrelated events.
  6. Fire-and-forget from the listener. Chrome's listener signature isn't async-friendly — we never await from inside it. Push to the queue, kick off the drain, return synchronously.

Dependency injection, lightly

The listener takes tabGroupManager as a parameter instead of importing a singleton. That keeps it testable (pass a fake) and makes the wiring visible in background.ts's composition root.

Try this
In the service worker's DevTools, paste: chrome.tabGroups.onCreated.addListener(g => console.log('GROUP', g)). Create a tab group in any window — watch the raw event fire. That's the signal our listener wraps.
Part 07 — Runtime

Calling Chrome APIs from TypeScript

The Chrome API surface is vast. TypeScript types make it tractable. Two calling styles coexist — always prefer Promises. 

// Modern, preferred
const groups = await chrome.tabGroups.query({});

// Legacy — still valid, harder to compose
chrome.tabGroups.query({}, (groups) => { /* ... */ });

Walkthrough: TabGroupManager.getGroup

Manager methods wrap Chrome APIs with logging, retries, and fallbacks. getGroup is a small example that shows all three ideas at once.

// src/lib/tabGroupManager.ts
async getGroup(groupId: number): Promise<chrome.tabGroups.TabGroup | null> {  // ①
  try {
    const group = await new Promise<chrome.tabGroups.TabGroup | null>((resolve) => {
      chrome.tabGroups.get(groupId, (g) => resolve(g || null));               // ②
    });
    if (group) return group;                                                  // ③

    // Fallback: scan every window. Useful after window crash/restore.
    const windows = await chrome.windows.getAll();                            // ④
    for (const window of windows) {
      const groups = await chrome.tabGroups.query({ windowId: window.id });
      const match = groups.find(g => g.id === groupId);
      if (match) return match;
    }
    return null;                                                              // ⑤
  } catch (error) {
    this.logger.error('getGroup:failed', { groupId, error });
    return null;                                                              // ⑥
  }
}
  1. Nullable return type. Promise<TabGroup | null> — not Promise<TabGroup>. Chrome windows can close between an event firing and the handler reading the group, so null is a real outcome callers must handle at the type level.
  2. Callback-to-promise bridge. chrome.tabGroups.get here is used in its legacy callback form, wrapped into a promise so the rest of the method can await. The || null normalizes an undefined callback argument.
  3. Happy-path fast exit. 95% of calls find the group on the first try. We only descend into the fallback when the direct lookup misses.
  4. Fallback: brute-force scan. Why do this at all? Because Chrome has edge cases — window just restored, group just moved — where get(groupId) returns nothing even though the group exists in some window. Querying every window is O(windows) but windows.length is typically small.
  5. Genuine miss. If the scan also fails, the group is really gone. Return null; the caller decides whether that's an error or a no-op.
  6. Exceptions become null + a log line. A throw from chrome.* (rare, but possible) is indistinguishable to callers from "not found." One fewer branch at the call site, one more log entry to debug from.
Key idea
Callers of any Chrome API should assume the object may have disappeared by the time the query resolves. Types with | null force the caller to face it.
Part 08 — Runtime

Managers and composition

Past a few hundred lines of background logic, classes that bundle related state win over loose functions passing dictionaries. This project uses five managers, wired at startup. 

// src/background.ts
async function initializeManagers() {
  storage = new StorageManager();
  await storage.initialize();

  bookmarkManager = new BookmarkManager(storage);
  syncEngine      = new SyncEngine(storage, bookmarkManager, null!);  // circular dep
  tabGroupManager = new TabGroupManager(syncEngine, storage);
  Object.assign(syncEngine, { tabGroupManager });                     // patched after

  snapshotManager = new SnapshotManager(storage, bookmarkManager);

  initializeTabGroupListeners(tabGroupManager, workerStartTime);
  initializeTabListeners(tabGroupManager);
  initializeBookmarkListeners(bookmarkManager);
}
ManagerResponsibility
StorageManagerPersist settings + history to chrome.storage.sync; hold runtime mappings in memory.
BookmarkManagerCreate/update bookmark folders that represent tab groups.
SyncEngineOrchestrator: given a group, make the bookmark folder match. Queues ops.
TabGroupManagerRead/mutate Chrome tab groups (e.g. move across windows).
SnapshotManagerCreate/restore point-in-time snapshots of a group.

Why classes?

State that belongs together — a sync queue, a map of move-guards, a hash of last-seen contents — is easier to pass around as a single object than as a bag of loose parameters. Constructors make dependencies explicit: a SyncEngine that needs storage and bookmarks takes both — nothing reaches into globals.

About that circular dep
SyncEngine needs TabGroupManager, which needs SyncEngine. We resolve it by constructing one with a null! placeholder, then patching in the real reference after both exist. It's not pretty — but it's honest. Circular deps happen when the domain has mutually recursive concerns, and the fix is usually "construct one, patch in the other," not "rearrange the domain."
Part 09 — Runtime

Popup ↔ background messaging

The popup and the service worker are separate processes. They talk via Chrome's message ports.

Walkthrough: popup-side initializeApp

When the toolbar icon is clicked, Chrome loads popup.html, which loads popup.js, which runs this. The React app does not mount until the service worker replies PONG.

// src/popup.tsx
async function initializeApp() {
  try {
    const port = chrome.runtime.connect({ name: 'popup' });    // ①

    await new Promise<void>((resolve, reject) => {
      const timeout = setTimeout(() => {                       // ②
        reject(new Error('Service worker connection timeout'));
      }, 5000);

      port.onMessage.addListener((message) => {
        clearTimeout(timeout);
        if (message.type === 'PONG')        resolve();          // ③
        else if (message.type === 'NOT_READY') reject(new Error(message.error));
        else                                reject(new Error('Unexpected response'));
      });

      port.onDisconnect.addListener(() => {                     // ④
        clearTimeout(timeout);
        if (chrome.runtime.lastError) reject(new Error('Failed to connect'));
      });

      port.postMessage({ type: 'PING' });                       // ⑤
    });

    const container = document.getElementById('root')!;
    createRoot(container).render(                               // ⑥
      <React.StrictMode>
        <ErrorBoundary><App /></ErrorBoundary>
      </React.StrictMode>
    );
  } catch (error) {
    document.body.innerHTML = `<div>Failed to Initialize…</div>`; // ⑦
  }
}
initializeApp();
  1. Open a named port. chrome.runtime.connect({ name: 'popup' }) fires chrome.runtime.onConnect in the service worker. The name is our convention so the worker can distinguish popup ports from any other context.
  2. Timeout as a reject path. A cold service worker may take a moment to wake. Five seconds is generous; anything longer and we'd rather show an error than spin.
  3. Two possible replies. PONG means managers are wired and it's safe to render. NOT_READY carries an error string the UI can display. Anything else means a protocol mismatch — treated as an error.
  4. Disconnect detection. If the service worker dies before responding, we never receive a message — onDisconnect fires instead. Without this handler, only the timeout would save us.
  5. Send ping last. Listeners must be attached before posting, otherwise a fast PONG could race past a not-yet-registered listener. Order matters.
  6. Render only on success. createRoot(...).render(...) mounts the React tree. Because we awaited the handshake, App is guaranteed to find a live service worker on its first message.
  7. Error fallback. On any rejection we replace document.body with a plain-HTML error card and a Retry button. No React involved — React itself might be why we failed.

Walkthrough: the App.tsx init effect

The popup side has a second handshake inside React. The PING done by initializeApp only proves the worker is alive; this useEffect is what builds the message-proxy managers the UI uses.

// src/components/App.tsx
export default function App() {
  const [isInitialized, setIsInitialized] = React.useState(false);  // ①
  const [error, setError] = React.useState<Error | null>(null);
  const [managers, setManagers] = React.useState<{ /*…*/ } | null>(null);

  React.useEffect(() => {                                           // ②
    const init = async () => {
      try {
        const port = chrome.runtime.connect({ name: 'popup' });     // ③
        await new Promise<void>((resolve, reject) => {
          const timeout = setTimeout(() => { cleanup(); reject(/*…*/); }, 5000);
          const messageHandler = (m: { type: string; error?: string }) => {
            if (m.type === 'PONG')      { cleanup(); resolve(undefined); }
            else if (m.type === 'NOT_READY') { cleanup(); reject(new Error(m.error)); }
          };
          const cleanup = () => {                                   // ④
            clearTimeout(timeout);
            port.onMessage.removeListener(messageHandler);
          };
          port.onMessage.addListener(messageHandler);
          port.postMessage({ type: 'PING' });
        });

        const storage: StorageManager = {                           // ⑤
          getSettings: async () => /* sendMessage('GET_SETTINGS') */,
          updateSettings: async (s) => /* sendMessage('UPDATE_SETTINGS', s) */,
          /* … */
        } as StorageManager;
        // bookmarkManager, syncEngine: same shape — each method = one sendMessage.

        setManagers({ storage, bookmarkManager, syncEngine, tabGroupManager });
        setIsInitialized(true);                                     // ⑥
      } catch (error) {
        setError(error as Error);                                   // ⑦
      }
    };
    void init();
  }, [logger]);                                                     // ⑧
  // … render branches on isInitialized / error / managers
}
  1. Three state atoms. isInitialized gates rendering, error triggers the error UI, managers is what child components consume. Splitting them makes each render branch trivial to check.
  2. useEffect for side effects after mount. React's contract: the function body is pure (builds the JSX), effects run after paint. Messaging Chrome is a side effect, so it goes here.
  3. Second PING — why? The popup.tsx handshake proved the worker was awake at that moment. Between then and now React has mounted App; the worker might have died and been restarted. This PING re-establishes the invariant for the lifetime of this component.
  4. Cleanup closure. Unlike popup.tsx, this handler removes its own listener when resolved. That matters because useEffect can re-run; leaking listeners onto the port would multiply handlers on every retry.
  5. Proxy managers. The popup doesn't instantiate real StorageManager classes — the real ones live in the service worker. Each method here is a thin chrome.runtime.sendMessage wrapper that matches the shape of the real manager. Components can't tell the difference.
  6. State flip = re-render. setManagers and setIsInitialized schedule a React render. Now the guard if (!isInitialized) return <Loading/> in the JSX becomes false and the real UI appears.
  7. Error branch. Any throw from the async init surfaces here as state, which a sibling branch of the JSX renders as an error card with a retry button (handleRetry, not shown).
  8. Dependency array. [logger] is effectively [] because logger is a singleton — the effect runs once per mount. A retryCount state is also added so handleRetry can force the effect to re-run.

Service worker side

// background.ts
chrome.runtime.onConnect.addListener((port) => {
  if (port.name !== 'popup') return;
  port.onMessage.addListener(async (msg) => {
    if (msg.type === 'PING') {
      const ready = await ensureInitialized();
      port.postMessage(ready ? { type: 'PONG' } : { type: 'NOT_READY' });

});

Two messaging APIs

  • chrome.runtime.connect → Port — long-lived, bidirectional, stays open while the popup is open. Used here for the handshake.
  • chrome.runtime.sendMessage + onMessage — fire-and-forget, one-shot. Used by every proxy manager method in App.tsx.

Persistence — storage.sync vs. bookmarks

Chrome gives extensions several places to store data. Picking wrong causes quota errors or lost state. This extension uses two storage surfaces by design.

chrome.storage.sync

Small, cross-device, synced via Google account. Good for settings. Strict limits: 100 KB total, 8 KB per item, 120 writes/min.

await chrome.storage.sync.set({
  'state:settings': this.persistedState.settings,
  'state:history':  this.persistedState.syncHistory.slice(-50),   // bounded!

.slice(-50) caps history to the last 50 entries — deliberately staying below quota. Unbounded arrays in storage.sync are a classic bug.

The product insight: bookmarks are already synced by Google, have no meaningful size limit, and are user-visible. So bookmark folders become the canonical storage for tab-group contents.

Part 11 — Deep dives

The distributed sync model

This extension is a distributed system — multiple Chrome instances, each holding partial state, converging without a coordinating server. Understanding the model makes the trade-offs make sense.

The topology

┌─────────────┐ ┌──────────────────┐ ┌─────────────┐ │ Chrome on │ │ Google Bookmark │ │ Chrome on │ │ Laptop │◀─ sync ──▶│ Sync Service │◀── sync ──▶│ Phone │ │ (ext v1.2) │ │ (operated by G) │ │ (ext v1.2) │ └─────────────┘ └──────────────────┘ └─────────────┘ ▲ ▲ │ │ chrome.bookmarks.* chrome.bookmarks.*

Each Chrome talks to the Google bookmark sync service. Our extension talks only to chrome.bookmarks — Google handles replication, conflict resolution at the bookmark level, and offline queuing. We ride a distributed system rather than build one.

Two consistency gotchas

Bookmark IDs are local

The ID "42" on your laptop is a different bookmark on your phone. IDs are per-profile, assigned on arrival. Never persist an ID across devices. Identify bookmarks by stable content — folder title, or URL for leaf bookmarks. See src/lib/bookmarks/bookmarkQueries.ts's signature-based search.

Names-as-identifiers

Tab group titles are the stable identifier across devices. Two consequences:

  • Two groups with the same name share one bookmark folder (documented in docs/KNOWN_ISSUES.md).
  • Renaming a group could mean "delete the old folder, create a new one" — which would orphan the original and lose all bookmarks under it. Instead we rename the bookmark folder in place whenever the title changes. See Part 12, Algorithm 6.

Conflict resolution: last-write-wins

If your laptop and phone modify the same group at the same moment, one wins at the bookmark sync layer. We don't fight it — we accept eventual consistency. The product guarantee is "your tab groups are eventually durable," not "every edit is preserved."

Design thread to pull on
Compare this approach to what Workona, Toby, and Session Buddy do: they run their own backends, maintain accounts, and implement server-authoritative conflict resolution. We trade features (merge conflict UIs, access logs, analytics) for a simpler product boundary: your data never leaves your browser. Every architectural decision in this repo ladders up to that trade.
Part 12 — Deep dives

Special algorithms

Five algorithms earn their keep here. None are individually exotic — their value is in combining to handle the messy realities of Chrome's event model.

1. Walkthrough: processSyncQueue

src/lib/sync/syncEngine.ts serializes sync operations with a 4-second gap and a quota-aware backoff. This is the single loop every bookmark write passes through.

// src/lib/sync/syncEngine.ts
private readonly SYNC_DELAY = 4000;
private syncQueue: QueuedSync[] = [];
private processingQueue = false;
private readonly MAX_QUEUE_SIZE = 100;

private async processSyncQueue() {
  if (this.processingQueue || this.syncQueue.length === 0) return;      // ①
  this.processingQueue = true;
  try {
    while (this.syncQueue.length > 0) {                                 // ②
      const item = this.syncQueue.shift()!;

      try {
        await this.syncGroupToFolder(item.name);                        // ③
      } catch (error) {
        const isQuotaError = error instanceof Error &&
          error.message.includes('MAX_WRITE_OPERATIONS_PER_HOUR');      // ④

        if (isQuotaError && (!item.retryCount || item.retryCount < this.MAX_RETRIES)) {
          this.syncQueue.push({                                         // ⑤
            ...item,
            retryCount: (item.retryCount || 0) + 1,
            timestamp: Date.now() + 60000
          });
          await new Promise(r => setTimeout(r, 60000));                 // ⑥
        } else {
          this.logger.error('syncQueue:syncFailed', { /*…*/ });         // ⑦
        }
      }

      if (this.syncQueue.length > 0) {
        await new Promise(r => setTimeout(r, this.SYNC_DELAY));         // ⑧
      }
    }
  } finally {
    this.processingQueue = false;                                       // ⑨
  }
}
  1. Single-runner gate. Same idea as the listener queue in Part 06, but here we drain in a while loop rather than self-rescheduling. Either style works; this one has less setTimeout overhead.
  2. Drain loop. Keep pulling items until empty. New pushes from queueSync that arrive mid-drain are picked up on the next iteration — they notice processingQueue === true and skip starting their own loop.
  3. Per-item try. One bad group can't abort the whole queue. The try is inside the loop so we always continue.
  4. String-match the error. Chrome signals quota exhaustion via a message substring. It's fragile, but it's the surface Chrome gives us. A typed error class would be better — file a Chromium bug if you're motivated.
  5. Re-queue with backoff metadata. The item goes back to the tail with an incremented retryCount. After MAX_RETRIES, step ⑤'s branch fails and we fall through to ⑦ — dropping the item permanently rather than thrashing.
  6. Cool-off sleep. A full minute of nothing. The quota is per-hour, so a minute is enough to let a few slots free up without stalling the whole drain. Other queued items wait — by design.
  7. Terminal failure logs only. Non-quota errors (or exhausted retries) become log lines. The queue continues with the next item.
  8. The 4-second heartbeat. Between every item, a fixed gap. chrome.storage.sync and bookmarks both quota at 120 writes/min — 4 s/item is 15/min, comfortably under even if each sync fans out to several writes.
  9. Release the gate in finally. Even if an exception escapes the loop (shouldn't, but belt-and-suspenders), the next caller can start a fresh drain.

2. Move-guards

Moving a group across windows fires onRemoved on the source window and onCreated on the target. Without a guard we'd treat the "remove" as a delete and wipe the bookmark folder. Each move installs a short-lived guard keyed by group name:

interface MoveGuard { sourceGroupId?: number; targetGroupId: number; expiresAt: number; }
private moveGuards: Map<string, MoveGuard> = new Map();
// before move:        this.moveGuards.set(name, { targetGroupId, expiresAt: Date.now() + 3000 });
// in remove handler:  if (guardActive(name)) return;   // ignore stale event

3. Content signatures for bookmark matching

Because bookmark IDs vary across devices (see Part 11), we match bookmarks by a signature: a hash of the URL list. When sync resumes on a new device, we find the right folder by comparing signatures, not IDs.

4. Walkthrough: resolveGroupName

Fifty lines in src/lib/utils/groupNameResolver.ts. Tiny, but it teaches a property-test discovery and a design decision.

// src/lib/utils/groupNameResolver.ts
export function resolveGroupName(title: string | undefined): string | null {
  if (title === undefined || title === null || title === '') {  // ①
    logger.logDecision('Group skipped due to missing title', /*…*/);
    return null;
  }
  if (title.trim() === '') {                                     // ②
    logger.logDecision('Group ignored due to whitespace-only title', /*…*/);
    return null;
  }
  return title;                                                  // ③
}
  1. Truly-empty → null. Chrome creates tab groups before the user names them, so title is initially undefined. Returning null tells every caller "this group isn't ready to sync yet" — no folder created, no mapping stored.
  2. Whitespace-only → null. The property-test origin: fast-check kept generating titles like " " which, if treated as valid, created folders named " " — hostile to bookmark sync. Rather than sanitize, the product decision was to ignore these groups entirely. Documented in .kiro/specs/tab-group-sync/requirements.md (Requirement 15).
  3. Valid → pass through. No trim, no lowercase, no normalization. The title is the cross-device identifier (Part 11); mutating it would break name-based matching.

Callers use the null return as the signal to skip work. Compare tabGroupListeners.ts: if (name === null) return. Three lines of this function replace what would otherwise be scattered if (title && title.trim()) checks in a dozen places.

5. Bounded history retention

// Ring buffer + age pruning
const MAX_ENTRIES = 200;
const MAX_AGE_MS = 7 * 24 * 60 * 60 * 1000;
history = history
  .filter(e => (now - e.timestamp) < MAX_AGE_MS)
  .slice(-MAX_ENTRIES);

Two independent cutoffs. Either condition can trigger eviction; whichever is tighter wins. This keeps storage.sync comfortably below quota without user configuration.

6. Title-change debounce with rename-in-place

Chrome fires chrome.tabGroups.onUpdated on every keystroke while the user types a group title. A naïve handler creates a bookmark folder for every intermediate string: type "splunk" and you get six folders — s, sp, spl, splu, splun, splunk. Two layers stacked make this correct for any typing pattern.

Layer 1 — listener-level debounce

// src/listeners/tabGroupListeners.ts
const UPDATE_DEBOUNCE_MS = 750;
const updateTimers = new Map<number, ReturnType<typeof setTimeout>>();
const latestUpdateByGroupId = new Map<number, chrome.tabGroups.TabGroup>();

chrome.tabGroups.onUpdated.addListener((group) => {
  latestUpdateByGroupId.set(group.id, group);           // ① always record newest

  const existing = updateTimers.get(group.id);
  if (existing) clearTimeout(existing);                 // ② reset on every event

  updateTimers.set(group.id, setTimeout(async () => {   // ③ fire only after silence
    updateTimers.delete(group.id);
    const latest = latestUpdateByGroupId.get(group.id);
    latestUpdateByGroupId.delete(group.id);
    if (!latest) return;
    await tabGroupManager.handleGroupUpdated(latest);   // ④ use settled value
  }, UPDATE_DEBOUNCE_MS));
});

chrome.tabGroups.onRemoved.addListener((group) => {
  const pending = updateTimers.get(group.id);           // ⑤ cancel on removal
  if (pending) { clearTimeout(pending); updateTimers.delete(group.id); }
  latestUpdateByGroupId.delete(group.id);
  // ... rest of removal logic
});
  1. Record, don't capture. If we captured group in the timer's closure, we'd sync the first keystroke's value ("s") instead of the last ("splunk"). Storing the latest separately and reading it at fire-time fixes this.
  2. Reset on every event. The debounce extends as long as events keep arriving — that's the whole point.
  3. Fire after silence. 750ms of no events means the user has likely stopped typing.
  4. Use the settled value. Whatever group happened to be in flight when the timer started is irrelevant.
  5. Cancel on removal. Otherwise the timer fires on a group that no longer exists, triggering a sync for a ghost.

Layer 2 — rename-in-place at the sync engine

Debounce alone is a liveness bet: if the user pauses to think for a second, 750ms fires too early. Without the second layer, pausing mid-typing still creates an intermediate folder. The fix: when a synced group's title changes, rename the existing bookmark folder instead of creating a new one. 

// src/lib/sync/syncEngine.ts
async handleGroupRenamed(
  oldName: string, newName: string, group: chrome.tabGroups.TabGroup
): Promise<void> {
  const oldMapping = await this.storage.getMapping(oldName);

  if (oldMapping?.folderId) {
    await chrome.bookmarks.update(oldMapping.folderId, { title: newName });   // ①
    await this.storage.updateMapping(newName, {                               // ②
      name: newName,
      folderId: oldMapping.folderId,
      currentGroupId: group.id.toString(),
      color: group.color,
      syncEnabled: oldMapping.syncEnabled,
      status: oldMapping.status,
    });
    await this.storage.removeMapping(oldName);                                // ③
    if (oldMapping.syncEnabled) this.queueSync(newName);                      // ④
  } else {
    await this.handleGroupUpdated(group);                                     // ⑤
  }
}
  1. Rename the folder, don't recreate. chrome.bookmarks.update(id, {title}) keeps the same folder ID and all child bookmarks — only the label changes.
  2. Migrate the mapping under the new name. The runtime mapping is keyed by name (Part 11), so a rename requires re-keying. We reuse the original folderId, preserving the bookmark chain.
  3. Drop the old mapping entry to avoid two entries pointing at the same folder.
  4. Queue a content sync only if the group was enabled. Disabled groups stay put.
  5. Fallback for first-title assignment. If there's no mapping for oldName, this is a group that's just being named — hand it to the create path.
Why both layers
Debounce alone can fire mid-typing and orphan a folder (if the sync engine ran the old create-new path). Rename-in-place alone is correct for any pause length, but calls chrome.bookmarks.update on every keystroke — wasteful. Stacking both: debounce reduces churn, rename-in-place guarantees correctness. A user could pause for 10 minutes between characters and still end up with a single folder named the final title, bookmarks intact.
Pattern
None of these six is a big algorithm. They're all small rules applied at the right event boundary. That's the style: instead of one clever data structure, many tiny invariants at the places where events cross system boundaries. Together they make the extension reliable.

Observability & debugging MV3

MV3's ephemeral worker makes stack traces lie. Good logging is the substitute. Every manager in this project funnels events through a structured Logger. 

// src/lib/utils/logger.ts (shape)
class Logger {
  info(event: string, data?: Record<string, unknown>) { /* ... */ }
  warn(event: string, data?: Record<string, unknown>) { /* ... */ }
  error(event: string, data?: Record<string, unknown>) { /* ... */ }
}

// Usage
this.logger.info('tabGroup:visible:handled', { groupId: group.id, title: group.title });
this.logger.error('sync:periodic:failed', { error: err.message });

Event names follow subject:verb:outcome. Payloads are structured JSON. That makes log lines filterable (tabGroup:*, sync:*:failed) and — importantly for MV3 — correlatable across worker restarts via timestamps and operation IDs.

OperationTracker

For multi-step ops, a tracker spans the lifetime: begin → progress → complete/fail. The service worker might die mid-op — the log will show a begin with no matching terminal event. That asymmetry is the clue.

Three MV3 debugging realities

Stale closures after worker restart

A variable captured in a listener at module load time is fresh. But the listener you registered during a previous worker's init is still referenced in chrome.runtime.* — and when it fires in a reincarnated worker, the captured module-level variable is whatever the new worker initialized it to. Don't rely on listener-level closures for mutable state — hold state in manager instances that get rebuilt on init.

Forgotten ensureInitialized

If a new event type arrives and you skip the init guard, the handler may run with undefined managers. Symptom: silent no-op or Cannot read properties of undefined. Every top-level listener must call await ensureInitialized() first.

Alarm firing before init

Chrome can fire an alarm the instant a worker wakes — before your async init has finished. The ensureInitialized pattern (Part 5) makes this safe. But if you put init in a setTimeout(fn, 5000), the alarm will beat it. Don't.

DevTools entry points

  • Service worker: chrome://extensions → service worker link under your extension. Console + breakpoints.
  • Popup: right-click the extension icon → Inspect popup. React DevTools works here.
  • Background events: the Errors button on the extension card lists uncaught exceptions across worker generations.
  • Storage: in the popup's DevTools, Application tab → Storage → Extension.
Worth internalizing
In a long-running MV3 app, the single most valuable debugging skill is reading logs across worker lifetimes. Not reading stack traces from a single run.

file:// URL sync — working around a Chromium limitation

Every Chromium browser silently drops file:// URLs from saved tab groups (bug #372978298). Bookmarks don't have this filter. This extension exploits that gap.

The one-line fix

The entire capture path was blocked by a single filter in bookmarkManager.ts:

// Before — dropped file:// URLs:
if (!tab.url.startsWith('http://') && !tab.url.startsWith('https://'))

// After — allows file://:
if (!isSyncableUrl(tab.url))   // http, https, OR file
src/lib/bookmarks/bookmarkManager.ts — the gate that was blocking file:// tabs

That's it for capture. The restore path in background.ts already opened any URL from bookmarks without filtering — so file:// bookmarks "just work" on restore, as long as the browser has "Allow access to file URLs" enabled for the extension.

The opener page fallback

When chrome.tabs.create({url: 'file://...'}) fails (usually because file access isn't enabled), the extension opens opener.html instead — a plain HTML page that shows the target path, setup instructions, and a "Try opening" button.

Tab has file:// URL │ isSyncableUrl() ── yes ──▶ canonicalize() ──▶ bookmark created │ chrome:// edge:// ── no ──▶ dropped (as before)
Why bookmarks work but saved groups don't
Chrome's "Saved Tab Group" feature uses a separate sync type that explicitly filters file:// at save time. Bookmarks are older, simpler infrastructure — they store any URL string. Our extension rides the bookmark path, bypassing the saved-group filter entirely.

Part 17 — File URLs & Search

Cross-machine path mapping — the flip-flop fix

File paths differ across machines: /Users/foo/Dropbox on Mac, /home/foo/Dropbox on Linux, C:\Users\foo\Dropbox on Windows. Without canonicalization, each machine writes its own path variant to bookmarks — creating duplicates that compound on every sync cycle.

Bidirectional mapping

Each machine has a config: a canonical prefix (what goes in bookmarks) and a local prefix (what opens on this machine). Mapping works in both directions:

// src/lib/utils/pathMapper.ts — pure functions, no state

// At capture time: local → canonical (for bookmarks)
canonicalize('file:///home/bar/Dropbox/ch1.html', config)
// → 'file:///Users/foo/Dropbox/ch1.html'

// At restore time: canonical → local (for opening)
localize('file:///Users/foo/Dropbox/ch1.html', config)
// → 'file:///home/bar/Dropbox/ch1.html'

Why pure functions, not a Manager

Every other domain concern in this codebase uses a Manager class. PathMapper deliberately breaks the pattern. Mapping is a stateless transformation — URL in, URL out. No lifecycle, no initialization, no Chrome API calls. It takes a config argument; it doesn't read storage. This makes it trivially testable with fast-check: 

// Correctness property: round-trip
fc.assert(fc.property(arbUrl, arbConfig, (url, config) => {
  expect(localize(canonicalize(url, config), config)).toBe(url);
}));

Storage design

Path mappings live under their own chrome.storage.sync key (state:pathMappings), separate from state:settings. This prevents the background's saveState() from overwriting mapping config — a bug that took three rounds to fix (the popup writes directly to sync storage, bypassing the background's in-memory state).

chrome.storage.sync: state:pathMappings → { ← separate key, never clobbered machines: { "mac": { rules: [{ canonical: "/Users/foo/Dropbox", local: "/Users/foo/Dropbox" }] }, "linux": { rules: [{ canonical: "/Users/foo/Dropbox", local: "/home/bar/Dropbox" }] } } } chrome.storage.local: machineId → "linux" ← per-machine, not synced
Lesson learned
Never nest transient config inside a storage object that another subsystem reads and writes. The background's saveState() does a full overwrite of state:settings — any field added by the popup gets silently wiped. Separate keys are the fix.

The search index

Why uFuzzy

uFuzzy is ~3KB, faster than Fuse.js (~25KB), and handles URL patterns well with its intraMode: 1 setting (allows gaps within words). The haystack is "${title} ${url} ${groupName}" — searching "react hooks" matches a tab titled "React Hooks Guide" in any group.

The restore dropdown

Each group has a cloud-download icon that opens a menu with four restore modes:

  • Restore all tabs — opens all bookmarks, skips already-open URLs
  • Add missing tabs — same as restore all (only missing)
  • Add file:// tabs only — opens local file bookmarks with path mapping
  • Replace group (destructive) — closes existing tabs, restores fresh

All restore logic runs in the popup context, not the service worker. This avoids MV3 service worker lifecycle issues where the SW dies before async restore operations complete. The popup has access to the same chrome.bookmarks and chrome.tabs APIs and stays alive as long as it's open.

Architecture decision
The original implementation sent messages to the background service worker for restore operations. The service worker kept dying mid-execution — a known MV3 limitation for long-running async work triggered by one-shot messages. Moving the logic to the popup fixed it completely, at the cost of duplicating some Chrome API calls between contexts. Pragmatism over purity.
